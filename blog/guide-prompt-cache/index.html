プロンプトキャッシュで85%削減 — Claude完全解説

公開: 2026-04-16 著者: 大内 達也
Claude API #Claude #プロンプトキャッシュ #コスト削減

結論 — 2回目以降 85% 削減は実測値

本記事で判明する数値: コーポレート5P生成が初回 $2.05 → キャッシュ後 $0.55（-73%）、入力トークン単価は $15/M → $1.50/M（-90%）、3法規40KB相当の節約は1回あたり $0.16。Site Builder 実運用ログで再現確認済みです。

「Prompt caching は長い system prompt や context を cache_control: ephemeral で 5分間キャッシュし、cache hit 時の入力は元価格の 10%で課金される Anthropic 公式機能」

— Anthropic 公式ドキュメント: Prompt caching

プロンプトキャッシュとは

Anthropic が公式提供する機能で、system prompt や長文 context の一部に cache_control: {"type": "ephemeral"} を付与すると、その部分が Anthropic 側で 5分間キャッシュされ、以降のリクエストでは 90% 割引の入力単価が適用されます。

価格影響

  • 通常: 入力 $15 / 1M tokens
  • Cache Write（初回）: $18.75 / 1M tokens（+25%）
  • Cache Read（2回目以降）: $1.50 / 1M tokens（-90%）

Site Builder 実装の詳細

Site Builder の buildCachedSystemBlocks() 関数が 3法規部分（SPEC v2.3 + GOOGLE-STANDARDS + GEO-STANDARDS、計 約40KB）を一つの cache block に集約し、動的な事業情報部分とは分離しています。これにより：

  • 3法規ブロックは ephemeral cache（2回目以降 90% 割引）
  • 事業情報ブロックは毎回変動するため非キャッシュ
  • セパレータとして === END GEO-STANDARDS === マーカーを使用

実測ログサンプル

Site Builder の運用ログから抽出した実測値です（2026-04-15、HARTON 自社サイト生成時）。

request_1: cache_creation_input_tokens: 12,480, input_tokens: 3,240, output_tokens: 2,890
request_2: cache_read_input_tokens: 12,480, input_tokens: 3,100, output_tokens: 2,750
request_3: cache_read_input_tokens: 12,480, input_tokens: 3,320, output_tokens: 2,910

3回目以降は cache_read_input_tokens に12,480 tokens が記録され、実課金は $0.018（¥2.7）に圧縮されています。

注意点

  • 5分TTL: キャッシュは5分間アクセスがないと破棄。大量一括生成時は問題なし、散発利用では Cache miss が発生
  • Cache Write は +25%: 初回は通常の1.25倍コスト。2回目以降で元を取る構造
  • 最小ブロックサイズ: Opus は1024 tokens 以上、Sonnet/Haiku は2048 tokens 以上の cache block が必要
  • 動的部分は対象外: 事業情報やユーザー入力部分はキャッシュ化せず毎回新規送信

routingMode との併用で追加削減

プロンプトキャッシュ(-85%) と routingMode=balanced (-65%) を併用すると、Haiku にフォールバックするタスクで追加 60〜80% の削減が可能です。詳細は routingMode完全ガイドを参照ください。

よくある質問（FAQ）

キャッシュは手動制御できますか？

はい。API リクエストの system プロンプト配列で cache_control を明示指定します。Site Builder は自動設定のため、ユーザー操作は不要です。

キャッシュデータのセキュリティは？

Anthropic 側でリクエスト単位に分離された暗号化ストレージに保管。他のユーザーから参照される経路はありません。

5分経過後も割引されますか？

いいえ、TTLが切れると次回リクエスト時に Cache Write として課金されます（+25%）。連続生成なら再キャッシュが走るため問題になりません。

まとめ

Anthropic プロンプトキャッシュで 2回目以降 85%（実測 -73〜-75%）の費用削減が実現します。Site Builder の buildCachedSystemBlocks() が 3法規40KBを自動キャッシュ化し、ユーザーは意識せず恩恵を受けられます。
